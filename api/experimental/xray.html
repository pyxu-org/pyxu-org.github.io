class XRayTransform(shape)[source]#

Bases: LinOp

X-Ray Transform (for \(D = \{2, 3\}\)).

The X-Ray Transform (XRT) of a function \(f: \mathbb{R}^{D} \to \mathbb{R}\) is defined as

\[\mathcal{P}[f](\mathbf{n}, \mathbf{t}) = \int_{\mathbb{R}} f(\mathbf{t} + \mathbf{n} \alpha) d\alpha,\]

where \(\mathbf{n}\in \mathbb{S}^{D-1}\) and \(\mathbf{t} \in \mathbf{n}^{\perp}\). \(\mathcal{P}[f]\) hence denotes the set of line integrals of \(f\).

Two class of algorithms exist to evaluate the XRT:

  • Fourier methods leverage the Fourier Slice Theorem (FST) to efficiently evaluate the XRT when multiple values of \(\mathbf{t}\) are desired for each \(\mathbf{n}\).

  • Ray methods compute estimates of the XRT via quadrature rules by assuming \(f\) is piecewise constant on short intervals.

XRayTransform computes samples of the XRT assuming \(f\) is a pixelized image/volume where:

  • the lower-left element of \(f\) is located at \(\mathbf{o} \in \mathbb{R}^{D}\),

  • pixel dimensions are \(\mathbf{\Delta} \in \mathbb{R}_{+}^{D}\), i.e.

\[f(\mathbf{r}) = \sum_{\{\mathbf{q}\} \subset \mathbb{N}^{D}} \alpha_{\mathbf{q}} 1_{[\mathbf{0}, \mathbf{\Delta}]}(\mathbf{r} - \mathbf{q} \odot \mathbf{\Delta} - \mathbf{o}), \quad \alpha_{\mathbf{q}} \in \mathbb{R}.\]
Parameters:

static init(arg_shape, origin=0, pitch=1, method='ray-trace', n_spec=None, t_spec=None, **kwargs)[source]#

Instantiate an X-Ray Transform.

Parameters:

  • arg_shape (NDArrayShape) – Pixel count in each dimension.

  • origin (Real, NDArray) – Bottom-left coordinate \(\mathbf{o} \in \mathbb{R}^{D}\).

  • pitch (Real, NDArray) – Pixel size \(\mathbf{\Delta} \in \mathbb{R}_{+}^{D}\).

  • method ("ray-trace", "fourier") –

    How to evaluate the XRT.

    • "ray-trace" computes the XRT via a ray-marching method implemented using the Dr.Jit compiler.

    • "fourier" computes the XRT using NUFFT methods. (See NUFFT.)

  • n_spec (NDArray) –

    • If method = "ray-trace": (N_l, D) ray directions \(\mathbf{n} \in \mathbb{S}^{D-1}\).

    • If method = "fourier": (N_n, D, D) reference frame specifiers \(\{\mathbf{n},\mathbf{u},\mathbf{v}\} \in \mathbb{S}^{D-1}\).

    The 1st column of each (D, D) orthogonal sub-matrix \(\mathbf{U}\) denotes the XRay line directions \(\mathbf{n} \in \mathbb{S}^{D-1}\). Subsequent columns define the orientation of the parallel samples.

  • t_spec (NDArray, list(dict)) –

    • If method = "ray-trace": (N_l, D) offset specifiers \(\mathbf{t} \in \mathbb{R}^{D}\).

    • If method = "fourier":

      • non-uniform case: (N_t, D-1) coordinates \(\mathbf{s} \in \mathbb{R}^{D-1}\) such that \(\mathbf{t} = \mathbf{U}_{\mathbf{u}, \mathbf{v}} \mathbf{s}\).

      • uniform case: (D-1,) dictionaries of (a, b, M) triplets defining implicitly \(\mathbf{s} \in \mathbb{R}^{D-1}\) such that \(\mathbf{U}_{\mathbf{u}, \mathbf{v}} \mathbf{s}\).

  • kwargs (dict) – Extra keyword parameters forwarded to RayXRT or FourierXRT.

Notes

  • XRayTransform is not backend-agnostic: Operator methods of the instantiated object assume inputs have the same array backend as n_spec.

  • method="ray-trace" requires LLVM installed on the system. See the Dr.Jit documentation for details.

  • DASK inputs are currently unsupported.

  • XRayTransform is not precision-agnostic: fourier methods perform inner computations at the precision of n_spec, whereas ray-methods are always done in single-precision.
